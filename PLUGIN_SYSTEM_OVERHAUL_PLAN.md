## 1. Executive Summary

SonicJS ships a plugin **SDK** (`PluginBuilder`, hook system, plugin registry, DB-backed plugin tables, admin UI) that is *architecturally sound but incomplete in execution*. The single load-bearing defect: **plugins declare routes/middleware/hooks, but nothing generically mounts them into the running Hono app.** Every core plugin is instead hand-wired in `packages/core/src/app.ts`, each with a comment warning it "MUST be registered BEFORE `/admin/plugins`." Community plugins, scaffolded apps, and any plugin relying on `PluginBuilder.addRoute()` silently 404.

This single defect is the root cause of, or directly related to, most of the open plugin issues (#758, #621, #829, #834) and is exactly what the **Infowall v3 plugin architecture** (the May 2026 progress doc) solved in a sister fork. That fork proved — empirically, with 2,824 passing tests — a clean model: **synchronous `register(app)` at construction time, asynchronous `wireRegisteredPlugins()` on first request**, with capability-gated context, typed hook subscriptions, cron declarations, and singleton services.

This plan adopts that model for SonicJS, cross-checks it against how **Strapi** and **Payload** solve the same problems, and lays out a phased, test-first rollout: build the SDK core, fix mounting, migrate all ~22 core plugins onto the new pattern (email first, as the reference), and back the whole thing with real unit + integration + E2E coverage.

### 2.2 The `Plugin` interface (today)

```ts
interface Plugin {
  name: string; version: string; description?: string
  author?: { name; email; url }; dependencies?: string[]
  compatibility?: string; license?: string
  routes?: PluginRoutes[]; middleware?: PluginMiddleware[]
  models?: PluginModel[]; services?: PluginService[]
  adminPages?: PluginAdminPage[]; adminComponents?: PluginComponent[]
  menuItems?: PluginMenuItem[]; hooks?: PluginHook[]
  install?(ctx): Promise<void>; uninstall?(ctx): Promise<void>

  configure?(config): Promise<void>

```

### 2.3 How mounting *actually* works today (the smoking gun)

`app.ts` mounts every plugin **by hand**, in a strict order, each guarded by a comment:

```ts
// app.ts (excerpt, line numbers from current branch)
230  // Plugin routes - Security Audit (MUST be registered BEFORE admin/plugins to avoid route conflict)
237  // Plugin routes - AI Search (MUST be registered BEFORE admin/plugins to avoid route conflict)
249  // Plugin routes - OAuth Providers (MUST be registered BEFORE admin/plugins ...)
263  // Plugin routes - OTP Login (MUST be registered BEFORE admin/plugins ...)
271  // Plugin routes - Analytics (must be before /admin/plugins catch-all)
281  // Plugin routes - Stripe (must be before /admin/plugins catch-all)
288  app.route('/admin/plugins', adminPluginRoutes)   // ← generic plugins admin
290  app.route('/admin', adminUsersRoutes)            // ← bare /admin catch-all
```

**Consequences:**
- A plugin that *only* uses `PluginBuilder.addRoute()` (e.g. global-variables) is never mounted → 404 (**#758**).
- Scaffolded apps don't know they must hand-mount or regenerate the registry (**#621**).
- Adding a plugin means editing core `app.ts` — community plugins can't do this (**#829**, **#830**).
- Route order is positional and fragile; the `/admin` catch-all shadows anything mounted after it.

### 2.4 Honest assessment

- **Type safety:** good (TS + Zod in places).
- **Extension points declared:** good.
- **Extension points *wired*:** broken. Routes/middleware/models declared in the builder are collected but not applied generically.
- **Capability gating:** absent.
- **Cron for plugins:** absent (workflow plugin has a bespoke `scheduled_at`).
- **Typed hooks:** hooks are string-keyed; payloads are untyped.
- **Test coverage of the wiring:** absent (the exact area that's broken is the area with no integration tests).

## 3. The Problem, Stated Precisely

> **Plugins describe what they want (routes, hooks, services, crons), but the host never generically applies those descriptions, and where it does apply them it must do so by editing core source in a position-sensitive way.**

Five concrete failures fall out of that one root cause:

1. **No generic route mounting** — `addRoute()` is dead metadata for non-hardcoded plugins (#758).
2. **Position-sensitive hand-wiring** — `/admin` catch-all shadows late-mounted plugin routes; ordering is encoded as comments, not enforced.
3. **No first-class registration API for consumers** — users must edit core or hand-mount (#621, #829).
4. **No capability/permission boundary** — any plugin can do anything; no `ctx.email`-style scoping.
5. **Untyped hooks + no cron** — string hook names, untyped payloads, no scheduled-task surface.

And the meta-problem: **the broken parts have no integration/E2E coverage**, so regressions are invisible until a user hits a 404 in production.

### 4.2 The Infowall v3 plugin architecture (May 2026 progress doc)

A sister fork built and shipped a v3 plugin SDK across four PRs (PR-S1 types, PR-S2 register pipeline, PR-S3 hook-system singleton + wiring, PR-M), then migrated the **email plugin (PR-E)** as the first real consumer. Key claims, all reportedly green (2,824 tests passing, tsc clean):

- **Two-pass lifecycle:** `register(app)` (sync, mounts routes) → `onBoot` (async, wires env-dependent services after *all* plugins register).
- **Capability-gated context:** `ctx.email` throws `SonicCapabilityError` if the plugin didn't declare `email:send`. Eleven capabilities defined.
- **Typed hook subscriptions:** subscribing to `auth:registration:completed` yields the narrowed payload shape; TS rejects wrong field names at the subscribe site.
- **Cron surface:** `crons: [{ schedule: '*/15 * * * *', hookFamily: 'email-reconciliation' }]` in the manifest; an `on-cron-tick` handler filters by `event.hookFamily`.
- **Singleton services:** `email-service-singleton.ts` mirrors `hook-system-singleton.ts` (three-function API, throw-before-set), set eagerly in the app factory so cron handlers running outside HTTP context still get the service.

**The empirically-discovered bug that reshaped the design (critical for SonicJS too):**

> **Hono's SmartRouter locks after the first request.** Calling `app.route()` from inside a middleware that fires lazily on first request throws `Error: Can not add a route since the matcher is already built`.

The fix splits registration into two functions:

| Function | When | What |
|---|---|---|
| `registerPluginRoutes(app, plugins)` | **Synchronous**, at construction, inline in the app factory, **between the last core mount and the bare `/admin` catch-all** | Validate, topo-sort, call `plugin.register?.(app)` for each. Routes mount before the matcher builds and before `/admin/*` — no shadowing. |
| `wireRegisteredPlugins(ordered, byId, hostContext)` | **Async**, lazy on first request | Hook subscriptions, cron registration, `onBoot`. Needs per-request env. |

Contract: **`register()` must be synchronous**; a `Promise` return is rejected with a named error rather than hanging. Async work belongs in `onBoot`.

> **Why this matters for SonicJS directly:** SonicJS has the *exact same* `/admin` catch-all-after-plugins structure (see §2.3). Any SonicJS plugin that mounts routes lazily will hit the identical SmartRouter lock. The fix is fork-agnostic. We should align on it rather than discover it independently.

## 5. Reference Deep Dive — Strapi v5

> Verified against docs.strapi.io (v5) and `strapi/strapi` `main`.

### 5.1 The big picture
A Strapi plugin is **two independently-loaded programs sharing a name**: a *server* program (`./strapi-server` export) and an *admin* program (`./strapi-admin` export). They never share a runtime; the only contract is the plugin **id**. Everything is wired through a single mutable global `strapi` object — a **service-locator/registry**, not constructor DI. Plugins are **plain config objects, not classes** (there's still a `// TODO: validate plugin format` in the loader — no shape validation).

### 5.2 Server entry & lifecycle
```ts
// server/src/index.ts
export default () => ({ register, bootstrap, destroy, config,
  contentTypes, routes, controllers, services, policies, middlewares })
```
| Hook | Timing | Available |
|---|---|---|
| `register({ strapi })` | Before app setup — **no DB, no routes yet** | Register custom fields, content-type extensions, document-service middleware |
| `bootstrap({ strapi })` | After full setup, before HTTP listen | Everything — DB, all plugins' services, seed |
| `destroy({ strapi })` | Shutdown | Cleanup |

**Two-phase lifecycle (register → bootstrap) is the same shape Infowall's v3 uses** (register → onBoot). Strapi validates this pattern at industrial scale.

### 5.3 Discovery & registration
Three sources merged with `defaultsDeep`, filtered by `enabled`:
1. **Internal core plugins** — hardcoded, always on (`content-manager`, `upload`, `i18n`, …).
2. **npm plugins** — auto-discovered: Strapi scans `dependencies`, requires each `package.json`, keeps those with **`strapi.kind === 'plugin'`**. *No `config/plugins.js` entry needed.*
3. **Declared plugins** — `config/plugins.js`; a `resolve` path makes it a local plugin.

```js
// config/plugins.js
module.exports = ({ env }) => ({
  i18n: true,
  'my-plugin': { enabled: true, resolve: './src/plugins/my-plugin', config: { /* … */ } },
})
```

### 5.4 Extension points & namespacing
Each plugin is wrapped as a **module** under namespace `plugin::<name>` and distributed into typed registries. Accessors resolve **lazily by string at call time**:
```ts
strapi.plugin('my-plugin').service('my-service')   // plugin::my-plugin.my-service
strapi.plugin('my-plugin').controller('x')
strapi.plugin('my-plugin').contentType('thing')
strapi.plugin('my-plugin').config('key')
```
Isolation = **namespacing only**, not sandboxing. Any plugin can read/mutate any registry.

### 5.5 Three interception layers (a known footgun)
1. **Document Service middleware** (`strapi.documents.use(...)`) — v5-recommended, content-agnostic, registered in `register()`.
2. **DB/ORM lifecycle hooks** (`beforeCreate`, `afterUpdate`, … 16+ events) — *don't fire for raw Knex; bulk ops fire only the batch event.*
3. **Event hub** (`strapi.eventHub.emit/on`) — pub/sub that webhooks consume.

> **Lesson for SonicJS:** Strapi's biggest plugin footgun is *three overlapping ways to intercept*, where picking the wrong layer silently misses cases. SonicJS should expose **one** typed hook layer, not three.

### 5.6 Config validation
Per-plugin `config: { default, validator }` where `validator` is an **imperative throwing function** (not a schema). User config from `config/plugins.js` merged with per-env overrides via `defaultsDeep` (user wins).

### 5.7 Type safety — candid
Lifecycle and `config` are typed; **`services`, `policies`, `middlewares`, `registerTrads` are still `unknown`** at the entry boundary, and accessors are stringly-typed (no per-plugin service map). Malformed plugins fail late.

### 5.8 Takeaways for SonicJS
- ✅ **Two-phase lifecycle** (register/bootstrap) — adopt.
- ✅ **Auto-discovery via a `package.json` metadata key** — adopt the idea (but build-time on Workers; see §7).
- ✅ **Namespacing** for plugin-owned tables/services — adopt.
- ⚠️ **Service-locator with stringly-typed lookups** — *improve on this* with typed capabilities/services.
- ❌ **Three interception layers** — *don't* replicate; one typed hook system.

## 6. Reference Deep Dive — Payload CMS v3

> Verified against payloadcms.com/docs and `payloadcms/payload` `main` (template fetched verbatim).

### 6.1 The big picture — a plugin is a config transformer
```ts
export type Plugin = (config: Config) => Config | Promise<Config>
```
That's the **entire contract.** No register API, no lifecycle object. Real plugins are curried: `(options) => (config) => Config`. Plugins are composed as a **left-to-right fold** over the `plugins: []` array:
```
config₀ → pluginA → config₁ → pluginB → … → sanitize → init
```
Order is the *only* dependency mechanism — no `dependsOn`, no topological sort.

### 6.2 What a plugin can do
Anything the `Config` can express: push new collections/globals, locate-and-mutate existing collections' fields, add `endpoints`, merge into hook arrays, push admin component **path-strings**, register `jobs.tasks`, and **chain `onInit`**:
```ts
const prev = config.onInit
config.onInit = async (payload) => { if (prev) await prev(payload); /* seed */ }
```

### 6.3 The merge discipline (the core correctness rule)
Because every plugin folds over a shared object, the rule is **never clobber**:
- Arrays → guard `||= []`, spread, append.
- Objects → shallow-spread to preserve siblings.
- Functions (`onInit`) → capture & invoke the previous.

TypeScript guarantees *structure*, not *composition* — it won't catch a forgotten spread that drops a prior plugin's contributions.

### 6.4 The `disabled` ≠ removed insight (load-bearing)
The official template, when `disabled`, **keeps schema-affecting additions** (collections/fields) and only skips *behavioral* additions (endpoints, components, onInit). Why: schema follows config, so a toggle that dropped collections would generate a **destructive migration**.

> **Lesson for SonicJS:** a plugin's **schema contributions** and its **runtime behavior** have *different lifecycles*. Conflating enable/disable with create/drop-table is the failure mode to avoid. This directly informs how SonicJS `activate/deactivate` should (not) touch tables.

### 6.5 Hooks model — one coherent set
Four tiers, all server-only (stripped from client bundle): root (`afterError`), **collection hooks** (`beforeOperation → beforeValidate → beforeChange → afterChange → beforeRead → afterRead → beforeDelete → afterDelete → afterOperation → afterError`), **field hooks**, **global hooks**, plus auth hooks. Strongly typed and generic over the document type: `CollectionBeforeChangeHook<Post>`.

### 6.6 Jobs / queues / tasks
Built-in queue: **Task** (handler + `inputSchema`/`outputSchema` + `retries` + `schedule`), **Workflow** (ordered tasks, retry from failure point), **Job** (queued instance), **Queue** (named bucket). Execution is *decoupled from queuing* — bin script / `autoRun` cron / `GET /api/payload-jobs/run`. **A `schedule` alone runs nothing**; you need an execution mechanism. A plugin registers a task by pushing into `config.jobs.tasks`.

> **Lesson for SonicJS:** on Cloudflare Workers, the "execution mechanism" is the Cron Trigger / `scheduled()` handler. Plugin cron declarations must be *collected* and dispatched by a single `scheduled()` entry — mirroring Infowall's `on-cron-tick` filtering by `hookFamily`.

### 6.7 Admin UI extension — path-strings, not imports
RSC-based: config references components by **`package/subpath#ExportName`** strings (never imports React into the Node config), resolved via a generated **import map**.

### 6.8 Functional transformer vs. imperative register — the central tradeoff
| | Functional transformer (Payload) | Imperative register (Strapi / Infowall / proposed SonicJS) |
|---|---|---|
| API surface to maintain | None (the `Config` type *is* the API) | A register/builder API to design & version |
| Composability | Trivial fold; later plugins see earlier output | Explicit; needs ordering rules |
| Dependency resolution | Manual (array order only) | Can do topo-sort / `dependsOn` |
| Safety / scoping | None — any plugin can rewrite anything | Capabilities can scope what each plugin touches |
| Footgun | Forgetting to spread silently clobbers | Async-in-register / route-after-catch-all |
| Type safety | Structural, end-to-end | As good as the register API is designed |

### 6.9 Takeaways for SonicJS
- ✅ **Strong end-to-end types** + generated slug/document types — aspire to this.
- ✅ **`disabled` keeps schema** — adopt as the activate/deactivate rule.
- ✅ **One coherent, typed hook set** — adopt.
- ✅ **Jobs decoupled from execution; cron needs a runner** — adopt for Workers `scheduled()`.
- ⚠️ **Pure functional transformer** — *not* a fit for SonicJS (Hono route mounting is imperative and order/SmartRouter-sensitive; we need an explicit register step). But borrow the **composition discipline** and **immutability mindset**.

## 7. Design Synthesis — What SonicJS Should Borrow

| Decision | Source | Rationale |
|---|---|---|
| **Imperative `register(app)` + async `onBoot`** two-phase lifecycle | Infowall v3 + Strapi register/bootstrap | Hono mounting is imperative & SmartRouter-sensitive; a functional transformer can't express "mount before the matcher builds." Two phases cleanly separate sync route mounting from async env-dependent wiring. |
| **Sync register at construction; async wire on first request** | Infowall v3 (empirical SmartRouter fix) | Avoids `Can not add a route since the matcher is already built`. SonicJS has the identical `/admin` catch-all structure and will hit the identical bug. |
| **A single shared `mountPlugin()` / `registerPluginRoutes()` primitive**, position-aware | Infowall §12 Q1; replaces hand-wiring in `app.ts` | Removes the 7+ hardcoded plugin mounts and the "MUST be before /admin/plugins" comments. |
| **Capability-gated context** (`ctx.email` throws `SonicCapabilityError` without `email:send`) | Infowall v3 | Real isolation boundary that Strapi/Payload lack. Phase 1 vocabulary: `email:send`, `hooks.*`, `db:*`, `cron:register`, `cache:*`, etc. |
| **Typed hook subscriptions** with narrowed payloads | Infowall v3 + Payload typed hooks | Replaces today's string-keyed, untyped hooks. TS rejects wrong field names at subscribe site. |
| **Singleton services** (three-fn API, throw-before-set) for env-independent access | Infowall v3 (`hook-system-singleton.ts` pattern already in repo) | Lets cron/`scheduled()` handlers get services without per-request env. |
| **Plugin cron declarations** dispatched by one `scheduled()` runner | Payload jobs + Infowall cron | Workers cron triggers fan out to plugins by `hookFamily`. |
| **`disable` keeps schema; only `uninstall` drops tables** | Payload `disabled`≠removed | Prevents destructive migrations on toggle. |
| **Namespacing (`plugin::<name>`)** for plugin-owned tables/services | Strapi | Avoids collisions; enables introspection. |
| **`package.json` metadata key for discovery** (build-time registry on Workers) | Strapi `strapi.kind` | Workers has no runtime `fs`; keep the build-time `generate-plugin-registry.mjs` but formalize it and document it (#621). |
| **Explicit `plugins.register: Plugin[]` consumer API** | PR #829 | The user-facing entry; carries forward #829 but corrected for the SmartRouter + `disableAll` issues. |
| **Zod-parsed DB rows** at the `PluginService` boundary | PR #834 | Typed shapes in/out of D1; no loose JSON. |

**What we deliberately reject:** Strapi's three interception layers; Payload's pure no-register functional model (incompatible with Hono mounting); runtime filesystem autoload (no `fs` on Workers).

## 8. Target Architecture — SonicJS v3 Plugin SDK

### 8.1 The plugin definition

```ts
// New: definePlugin() — the authoring entry point
export const emailPlugin = definePlugin({
  id: 'email',
  name: 'Email',
  version: '3.0.0',

  // Declared, validated capabilities. ctx is gated to exactly these.
  capabilities: ['email:send', 'hooks.auth:subscribe', 'cron:register', 'db:email_log'],

  // Plugin-owned tables (namespaced plugin::email_*). Created on install, dropped on uninstall ONLY.
  schema: [emailLogTable],

  // SYNC. Mounts routes. No async, no Promise. Throws if it returns one.
  register(app, ctx) {
    app.route('/admin/plugins/email', emailAdminRoutes)
    app.route('/api/email', emailApiRoutes)
  },

  // ASYNC. Runs on first request, after ALL plugins have registered.
  async onBoot(ctx) {
    const email = getEmailService()                 // singleton, env-independent
    ctx.hooks.on('auth:registration:completed', async (payload) => {
      // payload is the narrowed, typed shape — no casts
      await email.send(welcomeEmail(payload.user))
    })
  },

  // Cron declarations, dispatched by the single scheduled() runner.
  crons: [{ schedule: '*/15 * * * *', hookFamily: 'email-reconciliation' }],
  async onCronTick(event, ctx) {
    if (event.hookFamily !== 'email-reconciliation') return
    await reconcileDeliveries(getEmailService())
  },

  // Lifecycle (DB/schema only; never touches routes)
  async install(ctx)   { /* create plugin::email_* tables */ },
  async uninstall(ctx) { /* drop plugin::email_* tables */ },
  // activate/deactivate flip status ONLY — never create/drop schema (Payload lesson)
})
```

### 8.2 Two-phase boot (the heart of the fix)

```
createSonicJSApp(config)
        │
        ▼
  registerPluginRoutes(app, plugins)        ← SYNC, at construction
        │   • validate + topo-sort by dependsOn
        │   • app.route('/admin/logs', …)   (core mounts)
        │   • for each plugin: plugin.register(app, ctx)   ← routes mount here
        │   • app.route('/admin', adminUsers)  ← catch-all registers AFTER plugins
        ▼
  [ Hono SmartRouter matcher builds on first request — now safe ]
        │
        ▼
  First HTTP request
        │
        ▼
  wireRegisteredPlugins(ordered, byId, ctx)  ← ASYNC, lazy, once
            • ctx.hooks subscriptions
            • cron registration
            • plugin.onBoot(ctx)
```

**Contract enforced in code:**
- `register()` returning a `Promise` → throw `PluginRegisterMustBeSyncError`.
- Plugin routes always registered **between** core mounts and the `/admin` catch-all → no shadowing.
- `wireRegisteredPlugins` runs exactly once (guarded flag), lazily, so per-request `env` is available.

### 8.3 New / changed files

| File | Purpose |
|---|---|
| `plugins/sdk/define-plugin.ts` | `definePlugin()` — typed authoring entry. |
| `plugins/mount.ts` | `registerPluginRoutes(app, plugins)` + `mountPlugin()` shared primitive (carries #829 forward, SmartRouter-safe). |
| `plugins/wire.ts` | `wireRegisteredPlugins()` — hooks, crons, onBoot. |
| `plugins/capabilities.ts` | Capability vocabulary + `SonicCapabilityError` + `makePluginContext(caps)` gating. |
| `plugins/hooks/catalog.ts` | Typed hook event catalog (names → payload types). |
| `plugins/singletons/*-singleton.ts` | Service singletons (email, hooks, …) mirroring existing `hook-system-singleton.ts`. |
| `plugins/cron.ts` | Cron collector + `dispatchCronTick(event)` for the Workers `scheduled()` handler. |
| `services/plugin-service.ts` | Hardened with Zod row parsing (#834). |
| `app.ts` | Remove 7+ hardcoded plugin mounts; call `registerPluginRoutes()` once. |
| `config` (`SonicJSConfig`) | `plugins: { register: Plugin[]; disableAll?: boolean }`; deprecate `autoLoad`/`directory` no-ops (#829). |

### 8.4 Capability vocabulary (Phase 1)

`email:send` · `hooks.auth:subscribe` · `hooks.content:subscribe` · `cron:register` · `db:<table>` · `cache:read` · `cache:write` · `media:read` · `media:write` · `http:fetch` · `admin:menu`. (Payment/queue capabilities deferred to Phase 2 — matches Infowall §12 Q2's open question.)

### 8.5 Hook event catalog (typed, reconciled)

Reconcile the currently-declared events down to *dispatched* ones, typed. Content lifecycle (6, already firing) + auth events (`auth:registration:completed`, `auth:password-reset:requested`, `auth:password-reset:completed`). Each name maps to a payload interface; `ctx.hooks.on(name, handler)` infers the payload. **Action item:** audit declared-but-never-fired events and either fire or remove them (Infowall §12 Q3 — "ten events still declared-but-not-fired").

## 9. Phased Implementation Plan

Each phase is independently shippable and test-gated. **No phase merges without its tests green.**

### Phase 0 — Baseline & safety net (no behavior change)
- Add **integration + E2E tests that capture today's behavior** (which core plugin routes resolve, which 404). This is the regression guardrail.
- Land **#834** (Zod-parse `PluginService` rows) — pure hardening, no behavior change.
- **Exit:** full suite green; documented baseline of working vs. 404 plugin routes.

### Phase 1 — The mount primitive (fixes #758, #829, #621)
- Implement `plugins/mount.ts`: `registerPluginRoutes(app, plugins)` + `mountPlugin()`, **sync, position-aware**, inserted between core mounts and the `/admin` catch-all in `app.ts`.
- Add `plugins.register: Plugin[]` to `SonicJSConfig`; deprecate `autoLoad`/`directory` as documented no-ops.
- Enforce **`register()` must be sync** (reject Promise with named error).
- Remove the hand-wired plugin mounts from `app.ts` *one at a time*, replacing with `register()`.
- **Resolve #829's blocker:** define `disableAll` semantics explicitly — `disableAll` skips *core* plugin mounting; `register`-ed user plugins still mount unless individually disabled — and **make docs match code** (the @furelid review item). Add a test asserting the exact semantics.
- **Exit:** global-variables routes resolve (#758 closed); a custom plugin passed via `plugins.register` mounts with zero core edits; #829 unblocked; #621 docs updated.

### Phase 2 — Two-phase wiring + typed hooks
- Implement `wireRegisteredPlugins()` (`plugins/wire.ts`): lazy first-request, once-guarded; runs hook subscriptions + `onBoot`.
- Ship the **typed hook catalog** (`plugins/hooks/catalog.ts`); migrate the hook system to typed `on(name, handler)`.
- Keep the legacy `registerPlugins()` wrapper as a back-compat shim for existing tests.
- **Exit:** a plugin can subscribe to `auth:registration:completed` with a typed payload; wiring runs exactly once; legacy hook tests still pass.

### Phase 3 — Capabilities + singleton services
- Implement `plugins/capabilities.ts` + `makePluginContext()` gating; `SonicCapabilityError`.
- Introduce service singletons (email first) mirroring `hook-system-singleton.ts`.
- **Exit:** accessing `ctx.email` without `email:send` throws, caught by a unit test; cron-context (no request env) can still reach the singleton.

### Phase 4 — Cron surface
- `plugins/cron.ts`: collect `crons[]` from plugins; `dispatchCronTick(event)` wired into the Worker `scheduled()` handler; filter by `hookFamily`.
- **Exit:** a plugin's `onCronTick` fires on schedule in a Miniflare/integration test; graceful skip when credentials/bindings absent.

### Phase 5 — `definePlugin()` + Email reference migration (PR-E equivalent)
- Ship `definePlugin()`.
- Migrate the **email plugin** as the reference (it touches capabilities, hooks, cron, singleton, routes — if it holds for email it holds for all). Close the `reset_link` leak; add `email_log` observability.
- Fix **magic-link** (`c.env.plugins?.get('email')` → `getEmailService().send()`).
- **Exit:** email plugin runs fully on v3; documented as the template for the rest.

### Phase 6 — Migrate remaining core plugins (~20)
- Mechanically migrate each core plugin to `definePlugin()` using email as the template. Batch by risk: leaf plugins (hello-world, code-examples, testimonials) first; auth-adjacent (otp, oauth, user-profiles) and stateful (stripe, ai-search, analytics, workflow) last.
- Delete the corresponding hand-wired blocks from `app.ts` as each lands.
- **Exit:** `app.ts` has **zero** hardcoded plugin mounts; all plugins flow through `registerPluginRoutes()`.

### Phase 7 — Docs, DX, and acceptance plugins
- Plugin development guide (#359), registry-generation docs (#621), FAQ/field-types accuracy (#471).
- Build **two greenfield acceptance plugins** to prove the contract end-to-end: **MCP plugin (#784)** and **Web Monitor (#577, exercises cron + services + http:fetch)**.
- **Exit:** a third party can author + register a plugin using only public docs, no core edits.

### Phase 8 — Marketplace foundation (downstream, optional)
- With a stable contract, tackle #360 (registry/discovery/versioning) then #367 (ratings). Out of scope for the "fix it once and for all" core effort but unblocked by it.

## 10. Testing Strategy

> **Principle:** the area that is broken today (generic mounting + wiring) is the area with no tests. Every phase ships unit **and** integration **and** (where user-visible) E2E coverage. Target: no plugin behavior is "verified" by manual clicking.

### 10.1 Unit tests (`packages/core/src/__tests__/plugins/`, Vitest)

| Area | Cases |
|---|---|
| `mount.ts` | route mounting; multi-route plugin; **position** (plugin routes register before `/admin` catch-all); priority ordering; empty-plugin no-op; **`register()` returning a Promise throws**; topo-sort by `dependsOn`; circular-dep error. |
| `capabilities.ts` | `ctx.email` throws `SonicCapabilityError` without `email:send`; allowed capability resolves; unknown capability rejected at validate. |
| typed hooks | `on('auth:registration:completed', …)` infers payload; wrong field name is a *type* error (type-level test via `tsd`/`expect-type`); priority ordering; cancellation; scoped `unregisterAll` on deactivate. |
| singletons | throw-before-set; set-once; same instance across calls. |
| cron | collector aggregates `crons[]`; `dispatchCronTick` filters by `hookFamily`; graceful skip on missing binding. |
| `plugin-service.ts` (#834) | Zod parse of plugin/stats/activity/hook/route rows; numeric coercion; **invalid settings payload rejected**; structured activity details. |
| `disableAll` semantics | exact matrix: core off + user on; per-plugin disable; docs-match assertion. |

### 10.2 Integration tests (in-process app via `createSonicJSApp`, Miniflare D1)

| Area | Cases |
|---|---|
| End-to-end mount | `createSonicJSApp({ plugins: { register: [testPlugin] } })` → `GET` the plugin route returns 200 (the #758 regression test). |
| global-variables | activate → `GET /api/global-variables` 200, `GET /admin/global-variables` 200, menu item present (the literal #758 repro). |
| Two-phase wiring | `onBoot` runs once on first request; hook fires with real payload against **real D1**, not mocks. |
| Lifecycle | install creates `plugin::*` tables; **deactivate does NOT drop tables**; uninstall drops them (Payload lesson, asserted). |
| Cron | scheduled tick dispatches to the right plugin; reconciliation writes back to `email_log`. |
| Capability denial | a plugin without `email:send` getting `ctx.email` surfaces `SonicCapabilityError` through the request path. |
| Dependency order | plugin B depending on A boots after A; missing dependency errors clearly. |

### 10.3 E2E tests (Playwright, `tests/e2e/`)

## 13. Risks & Open Questions

**Risks**
- **Hono SmartRouter lock** — the central technical risk; mitigated by the sync-register design (§8.2) and an explicit "register returning a Promise throws" guard + test.
- **Big-bang `app.ts` edits** — mitigated by removing hand-wires one plugin at a time, each behind a passing integration test (Phase 6).
- **Workers has no runtime `fs`** — no directory autoload; discovery stays build-time (`generate-plugin-registry.mjs`), which must be documented (#621) and ideally validated in CI.
- **Destructive migrations on toggle** — mitigated by the Payload rule: activate/deactivate never touch schema; only install/uninstall do (asserted in integration tests).
- **E2E skip debt (#414)** — new specs must run, not skip; otherwise the guardrail is illusory.

**Open questions (to settle before/early in Phase 1)**
1. **`mountPlugin()` ownership** — does the shared mount primitive live in `@sonicjs-cms/core` only, or is it designed to be reused by forks (Infowall)? (Infowall §12 Q1.) Recommendation: core, exported.
2. **Capability vocabulary scope** — are payment/queue/scheduled-fetch capabilities Phase 1 or Phase 2? (Infowall §12 Q2.) Recommendation: Phase 2.
3. **Hook event reconciliation** — which of the ten declared-but-never-fired events do we fire vs. delete? Align names *now* to avoid divergent vocabularies. (Infowall §12 Q3 — highest-value alignment item.)
4. **Legacy hook adapter lifetime** — keep the v1→v3 adapter until the last plugin migrates (end of Phase 6), then remove. (Infowall §12 Q4.)
5. **`disableAll` exact semantics** — **DECIDED (Phase 1):** `disableAll: true` disables *all* plugins — core **and** user `register`-ed. No plugin routes are mounted and plugin bootstrap (DB seeding) is skipped; only the core CMS routes remain. Rationale: matches the original in-code intent ("Disable all plugins including core plugins"), matches `bootstrapMiddleware` which already skipped plugin bootstrap on `disableAll`, and is the simplest mental model. Code and docs now agree (resolves the @furelid review mismatch). Encoded in `mount-integration.test.ts`.
